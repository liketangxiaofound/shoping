# 一、设计题目概述

在课程实验基础上，本项目实现了一套 **B2C 电子商务购物网站**，在完整购物流程（注册登录 → 浏览商品 → 购物车 → 下单支付 → 订单管理）之上，按课程要求扩展了 **大数据采集、数据分析、商品推荐** 及选做功能 **反爬虫侦测与应对**。

## 1.1 网站类型

**电子商务网站（购物类）**

## 1.2 核心功能要求对照

| 课程要求 | 本项目实现情况 |
|----------|----------------|
| 用户注册/登录 | ✅ `/api/auth/register`、`/api/auth/login`，JWT 鉴权 |
| 商品展示 | ✅ 首页列表、分类筛选、搜索、商品详情 |
| 订购/收费 | ✅ 购物车、结算、模拟支付、订单状态流转 |
| 三类角色 | ✅ 用户（customer）、销售人员（seller）、管理者（admin） |
| 大数据采集（3.2） | ✅ 登录/浏览/购买/操作四类日志落库 |
| 数据分析与推荐（3.3） | ✅ 用户画像、趋势预测、异常监控、排行榜、趋势图、两种推荐 |
| 附加功能（3.4） | ✅ 反爬虫侦测与应对（限流、蜜罐、人机验证） |

## 1.3 技术栈总览

| 层次 | 技术选型 |
|------|----------|
| 前端 | Vue 3 + Vite + Vue Router + Pinia + Element Plus + Axios |
| 后端 | Node.js + Express |
| 数据库 | PostgreSQL + Prisma ORM |
| 缓存/限流 | Redis（可选，未启动时内存兜底） |
| 对象存储 | 阿里云 OSS（商品图片） |
| 认证 | JWT + bcrypt 密码加密 |

# 二、需求分析报告

## 2.1 Step 1：业务场景与痛点分析

### 2.1.1 业务场景

典型 B2C 电商场景：平台方（管理者）管理卖家与全局报表；卖家维护商品目录与订单；消费者在站内完成浏览与购买。课程要求在此基础上引入 **大数据能力**，使平台不仅能“卖货”，还能 **采集行为数据 → 分析 → 预测 → 推荐 → 防护异常爬取**。

### 2.1.2 痛点（需解决的课题）

| 编号 | 痛点 | 影响 |
|------|------|------|
| P1 | 缺乏统一的用户行为数据沉淀 | 无法做画像与推荐 |
| P2 | 销售趋势与异常难以人工发现 | 运营响应滞后 |
| P3 | 商品信息易被爬虫批量抓取 | 带宽浪费、数据被竞品利用 |
| P4 | 多角色权限边界不清 | 安全风险与误操作 |
| P5 | 图片与订单并发场景下数据一致性 | 超卖、支付状态错误 |

## 2.2 Step 2：可能的解决方案

| 痛点 | 方案 A | 方案 B |
|------|--------|--------|
| P1 | 埋点 SDK + 消息队列异步写入 | **应用层服务同步写 PostgreSQL 日志表** |
| P2 | 外接 BI 平台 | **自研统计服务 + 管理后台可视化** |
| P3 | WAF 云服务 | **应用层频率限制 + UA 检测 + 蜜罐 + 人机验证** |
| P4 | RBAC 中间件 | **JWT 载荷 role + 路由级 `requireAdmin/requireSeller`** |
| P5 | 数据库事务 | **Prisma `$transaction` 扣库存与写订单** |

## 2.3 Step 3：可行性分析

综合考虑 **个人项目工期、部署成本、课程验收可演示性**，采用：

- **方案选择**：关系型库存日志 + Node 统计服务 + Redis/内存限流（不引入过重的大数据组件如 Flink/Kafka）。
- **可行性**：Prisma 迁移与种子脚本可快速复现演示数据；前后端分离便于分模块开发与测试。
- **风险**：数据量极大时 PostgreSQL 聚合变慢 → 可通过索引、按日汇总表扩展（见系统设计应急预案）。

# 三、系统设计报告

## 3.1 整体架构设计

采用 **前后端分离 + RESTful API** 架构。

```
┌─────────────────────────────────────────────────────────────┐
│                      浏览器 (Vue 3 SPA)                      │
│  用户端: Home / Cart / Checkout / Payment / ProductDetail    │
│  卖家端: SellerLayout → 商品/订单/日志/指标                   │
│  管理端: AdminLayout → 卖家/报表/分析/反爬虫                  │
└───────────────────────────┬─────────────────────────────────┘
                            │ HTTP /api/*  (Vite Proxy → :3000)
┌───────────────────────────▼─────────────────────────────────┐
│                   Express 应用层 (Node.js)                    │
│  中间件: CORS / JSON / antiCrawler / JWT auth                │
│  路由: auth, products, carts, orders, seller, admin,         │
│        analytics, recommendations, anti-crawler, upload      │
└───────┬─────────────────┬─────────────────┬─────────────────┘
        │                 │                 │
   PostgreSQL          Redis (可选)     阿里云 OSS
   (Prisma ORM)        限流/封禁/令牌    商品图片
```

### 3.1.1 部署架构（建议）

| 组件 | 建议部署 |
|------|----------|
| 前端静态资源 | 阿里云 OSS + CDN 或 Nginx |
| 后端 API | 云服务器 ECS / 轻量应用服务器 |
| PostgreSQL | 云数据库 RDS |
| Redis | 云 Redis 或同机部署 |

## 3.2 数据库设计（核心实体）

```

       ├── Order ── OrderItem ── Product
```

**角色枚举**：`admin` | `seller` | `customer`

## 3.3 模块设计

### 3.3.1 用户与认证模块

| 项目 | 说明 |
|------|------|
| 功能 | 注册、登录、获取当前用户 |
| 技术 | `bcryptjs` 哈希密码；`jsonwebtoken` 签发 24h Token |
| 接口 | `POST /api/auth/login`、`POST /api/auth/register`、`GET /api/auth/me` |

### 3.3.2 商品与交易模块

| 子模块 | 职责 |
|--------|------|
| 商品服务 | 分页列表、搜索、详情、分类 |
| 购物车 | 增删改数量、合并登录用户购物车 |
| 订单服务 | 创建订单（事务扣库存）、支付、取消、确认收货 |
| 上传服务 | 图片压缩（sharp）→ OSS |

### 3.3.3 卖家模块

- 路由前缀：`/api/seller/*`，中间件校验 `seller` 或 `admin` 角色。
- 商品与分类 CRUD 均绑定 `sellerId`，实现数据隔离。

### 3.3.4 管理模块

- 路由前缀：`/api/admin/*`，`requireAdmin`。
- 卖家管理、销售报表、分析看板、反爬虫管理。

### 3.3.5 大数据模块（课程重点）

```
┌──────────────────┐     ┌─────────────────────┐
│ dataCollection   │────▶│ PostgreSQL 日志表    │
│ Service          │     │ Login/Browse/Purchase│
└────────┬─────────┘     │ /Operation           │
         │               └──────────┬──────────┘
         │                          │
         ▼                          ▼
┌────────────────┐       ┌─────────────────────┐
│ userProfile    │       │ salesTrend /        │
│ Service        │       │ Anomaly / Ranking   │
└────────────────┘       └─────────────────────┘
         │                          │
         └──────────┬───────────────┘
                    ▼
           ┌─────────────────┐
           │ recommendation  │
           │ Service         │
           └─────────────────┘
```

**用户画像算法概要**：

- **地域**：优先解析订单收货地址；否则根据登录 IP 映射省份（`regionParser`）。
- **购买力**：按累计消费与订单数划分高/中/低/潜在。
- **偏好分类**：浏览停留时长 + 购买金额加权打分，取 Top5 类别。

**销售趋势预测**：对近 N 日销售额序列做滑动平均 / 线性外推，输出未来数日预测值及评估指标（MAE 等）。

**异常监控**：按日/周聚合销售额，与历史均值比较，偏离超过阈值标记异常。

**协同过滤**：构建用户-商品购买集合，计算用户间 Jaccard 相似度，聚合邻居购买商品得分排序推荐。

### 3.3.6 反爬虫模块

| 环节 | 设计 |
|------|------|
| 保护范围 | `GET /api/products/*`、`GET /api/recommendations/*` |
| 侦测 | 滑动窗口频率、10s 突发、可疑 User-Agent、蜜罐路径 |
| 应对 | 429 + `needVerify`；算术挑战换 `X-Crawler-Token`；多次违规 Redis/内存封禁 IP |
| 审计 | `CrawlerEvent` 表 + 管理后台查询/解封 |

**蜜罐路径**（不对前端暴露）：

- `GET /api/products/export-all-secret`
- `GET /api/products/catalog.json`

## 3.6 风险评估与应急预案（PM 视角）

| 风险 | 可能性 | 影响 | 应急预案 | 可行性 |
|------|--------|------|----------|--------|
| PostgreSQL 连接失败 | 中 | 全站不可用 | `.env` 检查；Docker 一键启动；健康检查 `/health` | 高 |
| OSS 配置错误导致图片 404 | 中 | 商品无图 | 种子数据使用已上传 URL；后台重新上传 | 高 |
| Redis 未启动 | 高 | 限流精度下降 | 服务自动降级为内存 Map，单实例演示可用 | 高 |
| 支付接口 400（日志 userId 错误） | 低 | 无法付款 | 事务内使用 `order.userId`；单测支付流程 | 已修复 |
| 爬虫误判正常用户 | 中 | 体验差 | 提高已登录用户阈值；前端自动弹窗验证；后台解封 IP | 高 |
| 日志表数据膨胀 | 低 | 查询变慢 | 按 `createdAt` 索引；定期归档；后续可加日汇总表 | 中 |
| JWT 泄露 | 低 | 账号被盗 | HTTPS 部署；短过期时间；敏感操作二次校验 | 中 |

## 4.3 核心功能实现说明

### 4.3.1 购物流程

1. 用户登录后浏览 `/home`，请求 `GET /api/products`。
2. 进入详情页触发 `POST /api/analytics/browse` 记录类别与停留时间。
3. 加入购物车 `POST /api/carts`，结算页创建订单 `POST /api/orders`（事务扣减库存）。
4. 支付页 `POST /api/orders/:id/pay` 更新状态为 `paid` 并写入 `PurchaseRecord`。

### 4.3.2 数据采集实现

在 `dataCollectionService.js` 中封装：

- `recordLogin`：登录成功时写入 `LoginLog`。
- `recordBrowse`：详情页离开或定时上报。
- `recordPurchase`：支付成功后按订单项写入 `PurchaseRecord`。
- `recordOperation`：卖家/管理员关键操作写入 `OperationLog`。

### 4.3.4 推荐系统实现

- **也买了**：查询购买/浏览过该商品的用户 → 统计其购买的其他商品频次排序。
- **猜你喜欢**：当前用户购买集合 vs 其他用户集合 → Jaccard → 加权推荐得分。
- **首页**：登录用协同过滤，未登录用热销 fallback。

### 4.3.5 反爬虫实现

- 中间件：`middleware/antiCrawler.js` 全局挂载，对保护路径调用 `antiCrawlerService.inspectRequest`。
- 前端：`utils/crawlerGuard.js` + `request.js` 拦截 429 自动弹窗验证并重试。
- 管理：`AntiCrawler.vue` 展示统计与解封。

## 4.5 项目反思（PM 视角）

1. **预估准确性**：大数据模块实际耗时高于预期，主因是聚合 SQL 与前端图表联调；通过种子脚本生成 160+ 订单样本后，分析页面才有可展示曲线。
2. **预案有效性**：Redis 降级内存方案在本地未装 Redis 时验证了反爬流程，符合设计预期。
3. **心得体会**：先固化 **日志表结构** 再开发分析接口，可避免返工；角色权限宜在路由层一次性拦截，而非每个 Controller 重复判断。
4. **改进方向**：可增加日汇总表提升报表性能；推荐可引入 ItemCF；反爬可对接滑动验证码服务。

### (4) 对 AI 辅助编程的思考（≥500 字）

**AI 生成的代码是否都能理解？**  
不能全部逐行理解，尤其是 Prisma 聚合与推荐算法中的集合运算。处理方式是：让 AI 用注释说明输入输出；对核心路径（下单事务、支付、日志写入）人工走读一遍；用 Postman/脚本跑通后再合并。

**是否有 AI 错误建议？如何识别？**  
有。例如曾建议使用不存在的 admin 订单接口、Vue2 的 `:visible.sync` 写法。识别方法：对照官方文档；以运行结果为准；对“大范围重写”保持警惕，优先小步 diff。

**AI 会取代初级程序员吗？**  
短期内不会完全取代。AI 擅长模式化代码，但 **需求澄清、架构取舍、联调排错、安全与数据一致性** 仍需人负责。初级程序员若只会复制 AI 输出而不验证，反而容易被替代；若能驾驭 AI，效率会显著提升。

**程序员的核心竞争力（新理解）**  
本次项目体会到核心竞争力在于：**问题分解、领域建模（电商+日志+分析）、质量意识（权限/事务/限流）、沟通与文档、持续验证**。工具会变，能交付可运行、可维护、可演示的系统的能力不会贬值。

**下次如何更好利用 AI？**  
提前写好接口契约与数据表；让 AI 按模块生成但人工做 Code Review；每个迭代配 1～2 条自动化测试；报告类内容由 AI 起草后按评分标准删减；敏感配置永不写入仓库。
